# manual-category-registry Specification

## Purpose

行動記録の手動カテゴリを永続的なレジストリとして管理し、直近使用順で提示することで記録時のカテゴリ選択を高速化する。カテゴリはあくまで表示ラベルであり、時間集計・ルール評価・パスワード解錠には影響しない。

## Requirements

### Requirement: 手動カテゴリの永続化と既定シード
システムは行動記録の手動カテゴリを永続テーブルに保持する SHALL。各カテゴリは一意の名前・最終使用時刻・使用回数を持つ SHALL。ストア初期化時に既定カテゴリ（昼食・休憩・移動・仮眠・運動・雑務・その他）をシードする SHALL。カテゴリは行動記録上の表示ラベルであり、時間集計・パスワード解錠そのものには影響しない SHALL。ただし例外として、`TIMELINE` ルール条件から一致対象として参照されたカテゴリは、その条件の評価（当日合計分数の集計）を通じて目標達成判定・ゲート解錠に影響しうる SHALL。カテゴリの改名・削除に相当する操作は提供しない（カテゴリ名が `TIMELINE` 条件の安定キーを兼ねるため）。

#### Scenario: 初回起動で既定カテゴリが存在する
- **WHEN** カテゴリを一度も記録していない状態で `GET /api/categories` を呼ぶ
- **THEN** 既定7語（昼食・休憩・移動・仮眠・運動・雑務・その他）がシード順で返る

#### Scenario: TIMELINE 条件から参照されたカテゴリは評価に効く
- **WHEN** ラベル「運動」の `TIMELINE` 条件がルールに存在する状態で、当日「運動」カテゴリの手動記録を追加する
- **THEN** その手動記録の時間が当日の `TIMELINE` 条件の合計分数に算入され、閾値到達で条件が met=true になりうる

### Requirement: 直近使用順のカテゴリ取得
`GET /api/categories` は登録済みカテゴリを最終使用時刻の新しい順で返す SHALL。一度も使用されていないカテゴリ（未使用の既定語を含む）は、使用済みカテゴリより後ろに、シード順で安定して並ぶ SHALL。

#### Scenario: 直近使ったカテゴリが先頭に来る
- **WHEN** 「通院」を記録した直後に `GET /api/categories` を呼ぶ
- **THEN** 「通院」が先頭に、続いて残りのカテゴリが並んで返る

### Requirement: 記録時のカテゴリ使用登録（upsert）
手動エントリ（離席／空き時間）の記録時にカテゴリ名が与えられたとき、システムはそれを登録する SHALL。既存の名前なら最終使用時刻を記録時刻へ更新し使用回数を1加算し、未知の名前なら新規に登録する SHALL。前後空白を除去した結果が空になるカテゴリ名は登録しない SHALL。

#### Scenario: 新規カテゴリが翌日以降も残る
- **WHEN** カテゴリ「買い物」を自由入力して離席を記録する
- **THEN** 「買い物」がレジストリに登録され、別日の記録ポップオーバーでもチップとして表示される

#### Scenario: 既存カテゴリの再使用で先頭へ浮上する
- **WHEN** 既に登録済みの「昼食」でもう一度記録する
- **THEN** 「昼食」の最終使用時刻が更新され、次回の `GET /api/categories` でより前方に並ぶ

#### Scenario: 空白のみのカテゴリは登録しない
- **WHEN** カテゴリ名が空文字または空白のみの状態で記録される
- **THEN** レジストリに新規カテゴリは追加されない
