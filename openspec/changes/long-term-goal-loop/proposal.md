## Why

長期目標（issue #54）には、日次で繰り返す固定ルールでは表現できない要望がある。「ボリュームアップシャンプーを使えば髪質が良くなるのでは」という**賭け**を立て、「3日後に前髪の写真を出す」「7日間、使用感を答える」という**答え合わせ**を仕掛け、1ヶ月後にその連なりを**沿革**として読みたい。さらに「30日待たないと最終レポートの姿が分からない」ことへの不満がある。

現状これらは置き場所すら無い。**進行中の目標には詳細画面が存在せず**、目標カードからの遷移は完走後のレポートただ一つ（`goal-report`: 進行中は API 409・UI 導線なし）。判断は操作の副作用として蒸発し、唯一 `goal-report` の閾値変更マーカーだけが「理由つきの判断」を残している。

この app の本質は **PDCA ループ**である。本変更はそれを比喩ではなく画面にそのまま出す。**Plan（賭け）と Check（答え合わせ）を一級のデータにし、日記と区別して沿革に載せる**。線引きは「大きさ」ではなく**「検証がぶら下がっているか」**という構造で決まるため、ユーザーが「これは大きい出来事か？」と迷う余地がない。

画面設計は既存 app の筋（**書く＝振り返りタブ／読む＝目標タブ**）に合わせ、詰まる場所（今日タブのゲート）で解決できるようにする。詳細は `ユーザーフロー.md` を唯一の受け入れ基準とする。

## What Changes

- **Plan / Check を新設**する。振り返りタブの目標コーナーで、Plan（短文の賭け）を書き、そこに Check（答え合わせ）を複数ぶら下げる。Check なしの Plan も作れる。
- **Check は独立した2軸**を持つ: **種類**（📷 写真を投稿する／💬 質問に答える）と **いつ**（単発／範囲）。種類が「いつ」を決めることはない（📷×範囲、💬×単発 も作れる）。
- **Check は解錠ゲートに合流**する。**単発Check は達成するまでロックを繰り越す**（強いコミット）。**範囲Check はその日限りのゲート**とし、サボった日は繰り越さず期間を過ぎたら消える（「その日の姿」は後から撮れないため）。
- **今日タブから直接答える**。ゲートで足止めされている場所で、写真提出／質問回答／理由つき取り下げができる。
- **通知**: その日はじめてダッシュボードを開いたとき、アプリ内トーストを**1回**出す。時刻でスケジュールしない（croner・OS トースト・pwsh は使わない）。
- **理由つき取り下げ**が唯一の脱出弁。Check 単体でも Plan ごとでも取り下げられ、理由は必須。取り下げた事実は沿革に残る。
- **⑤沿革ブロックを新設**し、Plan と Check の答え合わせを時系列に表示する。**日記は沿革に載せない**（④日記リーダーで参照する）。
- **BREAKING（spec 上）**: `goal-report` の「完走後のみ開ける」制約を撤廃し、**進行中でもレポートを開ける**（＝走行中プレビュー）。進行中カードのボタンは「レポートプレビュー」、完走後は既存の「レポートを開く」。
- **BREAKING（spec 上）**: `goal-report` の「ヘッダ＋4ブロック構成」を **5ブロック構成**に変更（⑤沿革を追加）。
- 走行中プレビューのため、①達成カレンダーで**「未到来」と「欠測＝未達」を区別**する。まだ来ていない日を未達成の黒星で埋めない。

## Capabilities

### New Capabilities
- `goal-plan-check`: 振り返りタブでの Plan（賭け）と Check（答え合わせ）の作成・保持。Check の2軸（種類×いつ）、1 Plan : N Check、Check なし Plan、理由つき取り下げを規定する。
- `goal-check-gate`: Check が解錠ゲートへ合流する挙動。単発＝繰り越し／範囲＝その日限りの使い分け、今日タブからの回答、初回オープン時のトースト通知を規定する。
- `goal-chronicle`: 沿革（⑤ブロック）の内容と並び。Plan と Check の答え合わせを時系列に載せ、日記を載せないことを規定する。

### Modified Capabilities
- `goal-report`: 「完走後のみ開ける」を撤廃し進行中でも開ける（走行中プレビュー）／「4ブロック構成」を「5ブロック構成」へ（⑤沿革）／①達成カレンダーで未到来を区別／③の最終日写真 CTA は完走後のみ。

## Impact

- **DB**: `goal_plan`・`goal_check`・`goal_check_result` の新規テーブル。マイグレーション追加（`server/src/db/migrations.ts`）。
- **解錠評価**: `server/src/rules/evaluate.ts` が、当該日に有効な Check を合成条件（`conditionKey='check:<id>'`）として `per_condition_results` へ AND 合流させる。latch・`is_final` スナップショットとの整合を規定。
- **画像保存**: 既存 `goal_image`（`goal-journal`）を写真Check の提出先に再利用。先指定キャプションで保存するため `goal-report ③` の Before/After へ自動流入する。
- **API / サービス**: Plan/Check の CRUD・回答・取り下げ、沿革取得を追加。契約スキーマ（`packages/contract/src/index.ts`）に型追加。
- **レポート**: `server/src/services/goals.ts` のレポート生成が進行中の目標を受け付ける（409 撤廃）。未到来日の表現を追加。⑤沿革を返す。
- **ダッシュボード**: 振り返りタブ（Plan/Check フォーム）、今日タブ（不足条件・回答・取り下げ・初回トースト）、目標タブ（レポートプレビュー導線・⑤沿革描画）。
- **デモモード**: `demo-seed.ts` に Plan/Check/回答/取り下げのサンプルを固定 day_key で追加し、`demo.test.ts` の期待値を更新（日数が関わる機能はデモモードで成果を再現するプロジェクトルール）。
- **e2e**: 既存 Playwright に、ユーザーフローの背骨を通す1本を追加。
