## Context

タイムラインの手動記録は現状「1区間1カテゴリ」で、`activity_log_entry(entry_type='MANUAL')` を1行作る（`addManualEntry`）。強調（`timeline-tracked-highlight`）は追跡中カテゴリ名の `Set` を持つだけで理由（閾値）を伝えない。記録ポップオーバーのチップは `chipCategories()` が直近使用順を先頭 `MAX_CHIPS=12` で打ち切るのみで、追跡カテゴリの固定表示や「もっと見る」は無い。

既存資産で再利用できる点が2つある。(1) `activity_log_entry` は AUTO セッションの同時オープン按分のために `n`（分母）と `credited_ms`（持ち分）を既に持ち、`coactive_group_keys` も持つ。(2) タイムラインのレーン描画 `layout()` は重なるブロックを列分割で並列配置する（作業グループの同時オープンがまさにこれ）。同時記録の「並列表示」はこの既存レイアウトに乗せられる。

`TIMELINE` ルール評価（`server/src/rules/evaluate.ts` の `case 'TIMELINE'`）は当日の一致 MANUAL の `SUM(end_at - start_at)` を秒換算して閾値比較している。

## Goals / Non-Goals

**Goals:**
- 追跡強調に「なぜ強調か」（採用中 `TIMELINE` 条件の当日実効閾値）を `≥N分` バッジで併記する（表示専用）。
- 記録ポップオーバーで追跡カテゴリを先頭固定・上限・「もっと見る」で提示し、複数カテゴリ選択に対応する。
- 1区間を N カテゴリで均等割同時記録し、区間全体を占める並列ブロックとして表示、各カテゴリを `区間長 ÷ N` で按分計上する。
- `TIMELINE` 評価が同時記録を持ち分（按分後）で算入する。

**Non-Goals:**
- 作業概況ドーナツ／円グラフへ自己申告時間を加える集計・凡例・色割当（別 issue）。本変更後もドーナツは AUTO 作業のみ。
- カテゴリの改名・削除、離席閾値 `away_min_seconds` の変更。
- 同時記録メンバーごとに区間（start/end）を個別編集して食い違わせる高度な編集 UI。

## Decisions

### D1: 同時記録グループを `co_record_group_id` + 既存 `n` で表す
`activity_log_entry` に nullable な `co_record_group_id INTEGER` を追加する（加算的移行、バックフィル不要）。同時記録の各メンバー行は同一 `co_record_group_id` を持ち、`start_at`/`end_at` は区間全体で共通、`n` にグループ構成数を格納する。単独記録は従来どおり `co_record_group_id=NULL`・`n=1`。
- **なぜ**: AUTO の `n`/`credited_ms` 按分パターンをそのまま MANUAL へ広げられ、意味が一貫する（`n` = 持ち分の分母）。
- **代替案**: 「同一 start/end の MANUAL をグループとみなす」導出方式 → 独立に同区間へ入れた単独記録2件を誤って0.5按分してしまうため却下。明示グループIDが必要。

### D2: 持ち分は `(end_at − start_at) ÷ n` を読み取り時に算出（MANUAL は `credited_ms` を持たせない）
表示上の区間は等分せず区間全体のまま、時間を参照する側が `span ÷ n` を持ち分として使う。`TIMELINE` 評価の SQL を `SUM(end_at - start_at)` から `SUM((end_at - start_at) * 1.0 / n)` へ変更する（`n` は既定1なので単独記録・既存データの結果は不変＝後方互換）。グループから1件削除したら、残メンバーの `n` を新しい構成数へ更新して再按分する。
- **なぜ**: 単一の真実（`n`）から持ち分を導出でき、`credited_ms` の二重管理・ドリフトを避ける。既存の単独記録・過去データに影響しない。
- **代替案**: MANUAL にも `credited_ms` を焼き込む → 再按分（削除時）で更新漏れドリフトのリスク、却下。

### D3: 並列表示は既存 `layout()` に委ねる
同時記録メンバーは start/end が完全一致で必ず重なるため、既存の列分割レイアウトが自動で隣接列へ並べる。新規レイアウトコードは不要。`keyOf` は MANUAL を `m:<id>` で区別済みなので各メンバーが別列に載る。ブロック描画にバッジと（任意で）持ち分ラベルを足すのみ。
- **なぜ**: 「作業グループと同様の並列」という要件を追加ロジックなしで満たす。

### D4: 複数カテゴリ確定は一括 API で原子的に作成
`POST /api/timeline/:date/manual` を、単一エントリに加えて複数カテゴリ配列を受け付ける形へ拡張する（後方互換：単一 body はそのまま）。サーバーは1トランザクションで N 行を同一 `co_record_group_id`・`n=N` で作成し、途中失敗で部分作成しない。クライアント `openDraft` はチップをトグル選択（集合）＋自由入力の複数追加に対応し、確定時に一括送信する。
- **なぜ**: スペックの「途中失敗で部分状態を残さない（MUST NOT）」を満たすには単一トランザクションが必要。N 回 POST は非原子的で却下。

### D5: 閾値バッジ用データはサーバーの目標 payload で提供
`GET /api/goals` の practice（`conditionKey` が `timeline:*`）に、当日実効ルールで解決した `thresholdSeconds` を付与する。クライアントは `refreshTrackedCategories` を拡張し、カテゴリ名→最大閾値秒の `Map` を作ってブロック／チップのバッジ（`≥N分`）に使う。バッジは表示専用で評価・集計・解錠に非影響。
- **なぜ**: 閾値の権威はサーバーのルールセット。クライアントで rules を突き合わせるより、既に読む goals payload に載せる方が単純。
- **代替案**: クライアントが `/api/rules` を別途取得して突き合わせ → クライアント側ロジック増、却下。

### D6: チップの固定・上限・展開はクライアント表示層のみ
`chipCategories()` を、(a) 追跡カテゴリ（上限外・常に先頭固定）、(b) 残りを直近使用順に上限まで、(c) 超過分は「もっと見る」で展開、の3区分に再構成する。レジストリ API・永続化は変更しない。
- **なぜ**: 要件は提示順・可視性の問題であり、`manual-category-registry` の要件は不変。

## Risks / Trade-offs

- **[再按分のドリフト/削除時の更新漏れ]** → 削除・追加を同一トランザクションで残メンバーの `n` を再計算し、単体テストで 2→1 の再按分を検証する。
- **[`n` の二重の意味（AUTO 同時オープン数 / MANUAL 同時記録数）]** → どちらも「持ち分の分母」で意味が一致。移行コメントとスペックで明記する。
- **[後方互換]** → `n` 既定1・`co_record_group_id` NULL のため既存の単独 MANUAL と過去データは挙動不変。旧 body の単一 POST も受理し続ける。
- **[メンバーの時刻個別編集で span が食い違う]** → 本変更では span 個別編集を非目標とし、持ち分は各メンバーの `span ÷ n` で定義（食い違っても定義は成立）。必要なら別途対応。
- **[円グラフ未対応で片手落ちに見える]** → 非目標として明示。持ち分（`span ÷ n`）を評価と同じ規則で持つため、後続の円グラフ issue はこの持ち分をそのまま集計に使える設計にしておく。

## Migration Plan

1. 加算的移行で `activity_log_entry.co_record_group_id INTEGER`（nullable, 既定 NULL）を追加。バックフィル不要（既存は単独記録）。
2. サーバー：一括作成 API・`TIMELINE` 評価の按分 SQL・goals payload の `thresholdSeconds` 付与・削除時の再按分。
3. クライアント：ポップオーバー複数選択、チップ固定/上限/もっと見る、ブロック/チップの閾値バッジ、並列表示確認。
4. ロールバック：カラムは旧コードから無視され `n=1` 相当で単独動作するため安全。API は単一 body 後方互換。
