## Context

「30日チャレンジ」（goal-challenge / goal-report / goal-journal、2026-07-12 実装済み）は、目標の状態を保存せず現在の day_key と `start_day`/`end_day` の比較で `開始前 / 進行中 / 完走` を導出し、完走レポートは `unlock_evaluation.per_condition_results` に日ごとに焼き込まれた達成状況から集計する。時刻に依存する読み取り系は `todayKey(db, nowMs=Date.now())`・`listGoals(db, nowMs)`・`getGoalReport(db, id, nowMs)`・`daySummary(db, dayKey)` のように **既に時刻を外から渡せる形**になっている。

本番の解禁は `POST /api/password/reveal` → `revealPasswords(db, date)` で、指定日が**実際の記録にもとづき UNLOCKED のときだけ**パスワード生成コマンドを実行して返す（未達成では絶対に返さない・脱出弁なし）。つまり本番ゲートは「実達成に紐づく」設計で、クライアントが日付を渡しても達成そのものは捏造できない。

このコンテキストの上に、issue #20 の「読むだけツアー型（A案）」デモを、本番データ・本番ゲートに一切干渉させずに載せる。

## Goals / Non-Goals

**Goals:**
- サンプル目標＋30日分の記録を投入し、開始前 → 進行中 → 完走 → 完走レポートを仮想日付ジャンプで体験できる。
- 日付操作・サンプルデータが本番データにも本番の解禁判定にも**構造的に到達しない**ことを保証する（規約や注意書きではなく、経路の分離で担保）。
- 既存の集計・レポート描画コード（`listGoals`/`getGoalReport`/`daySummary` とレポート UI）を再実装せずそのまま再利用する。

**Non-Goals:**
- ログイン／アカウント認証（#20 では「デモ切替」で十分）。
- デモ内での目標作成・削除・本物のパスワード生成（A案の閲覧専用方針で除外）。
- 習慣目標化（#21）・画像添付 Before/After（#22）・目標作成プレビュー。

## Decisions

### D1. 分離は「別インメモリ DB＋専用リード API」で行う（同一 DB 名前空間分離を採らない）

デモ用データは、本番 DB とは別の **インメモリ SQLite（`better-sqlite3` の `:memory:`）** にマイグレーションを適用し、サンプルを seed した使い捨てデータセットとして持つ。サーバは1つの常駐デモ DB を遅延構築してキャッシュし、`サンプルをリセット`／`デモ開始` で再 seed する（単一ユーザのローカルアプリなのでセッション多重化は不要）。

- **なぜ別 DB か**: 「本番データ無傷」「本番ゲートに絶対効かせない」を、注意書きではなく**経路の物理分離**で担保できる。デモ集計は本番 DB のコネクションに一度も触れない。
- **却下: 同一 DB＋`is_demo` フラグ名前空間分離** — 本番テーブルにデモ行が混ざり、集計・レポートの各クエリで漏れなく除外する責任が全所に散る。1箇所の漏れが本番汚染に直結するため、ガードレール要件に対して脆い。

### D2. 仮想「今日」はクライアント所有、デモ読み取りに毎回渡す（サーバに可変な時計状態を持たない）

上部バーの日付コントロールが持つ仮想 day_key を、デモの各リード要求に付けて送る。サーバは受け取った仮想 day_key をデモ DB の設定（tz・境界）で `nowMs` に変換し、**既存のサービス関数をデモ DB と仮想 now で呼ぶ**（`listGoals(demoDb, virtualNow)` / `getGoalReport(demoDb, id, virtualNow)` / `daySummary(demoDb, virtualDayKey)`）。これが issue 補足の「“今の日付”を差し替えられる作り＝仮想 now の中央注入」。

- **なぜクライアント所有か**: リロードやサーバ再起動で時計状態がずれない。サーバは「seed 済みデモ DB」と「純粋な read」だけを持ち、状態機械を増やさない。
- `＋1/＋7/＋30` は仮想 day_key を加算、`完走へ` は `end_day + 1` へ、`リセット` は `start_day − 1`（開始前）へ設定するだけ。

### D3. デモ API は読み取り専用の別ルータ（`/api/demo/*`）。reveal・書き込みルートを持たない

デモに必要な read だけを列挙した専用ルータを追加し、**デモ DB のみ**に束ねる:
- `POST /api/demo/reset` — デモ DB を（再）seed し、初期仮想 day_key（開始前＝`start_day − 1`）と目標概要を返す。
- `GET /api/demo/goals?now=<dayKey>` — 仮想日付での目標一覧（導出状態つき）。
- `GET /api/demo/goals/:id/report?now=<dayKey>` — 完走レポート4ブロック。
- `GET /api/demo/goals/:id/journal/:date` — 目標日記（サンプル記入済み）。
- `GET /api/demo/today?now=<dayKey>` — 仮想日付のサマリ＋**ダミーの解禁/パスワード表示**。

このルータは reveal・パスワード生成コマンド・本番 DB 書き込みのいずれの関数も import しない。today のデモ版はダミー値を返すだけで `revealPasswords` を呼ばない。→ **「本物のゲートに到達しない」が呼び出しグラフ上で成立**する。

- **却下: 既存エンドポイントに `demo=1&now=` を生やして本番 DB/デモ DB を分岐** — 全既存ハンドラにデモ分岐が漏れ込み、reveal 経路にも「デモなら…」の条件が必要になる。1つの分岐ミスがゲート破りになるため、専用ルータで経路自体を分ける。

### D4. サンプルは固定 day_key に seed し、集計が読むテーブルを直接埋める

seed は goal-report の design-brief の筋書きをそのまま使う（本人発案の既存サンプル）: 目標「メンタルを安定させる」、実践3つ（振り返りを書く / 明日のタスク登録 / 作業4時間）、期間 `2026-06-11`〜`2026-07-10`、達成 24/30（中盤に谷→後半持ち直し）、Day13 に閾値 4h→3h（理由「課題週間。ゼロにはしない」）、30日分の日記、Day1/Day30 の Before/After。

レポート②（時間推移）と①（達成カレンダー）が読むのは `unlock_evaluation.per_condition_results`（conditionKey/met/actualSeconds/thresholdSeconds を日ごとに焼き込み済み）と時系列スナップショットなので、**そのテーブルを直接 seed** すればよい（30日ぶんのゲート評価パイプラインを再現走行させない）。`goal`/`goal_practice`/`practice_threshold_change`/`goal_journal` も固定値で挿入。day_key を固定するため seed 側は `Date.now()` に依存しない。

- 仮想 day_key はこの固定期間の内外（`2026-06-10` 開始前 〜 `2026-07-11` 完走）を動くだけ。既存の導出ロジック（`today < start_day` → 開始前 …）がそのまま状態を返す。

### D5. フロントはデモ状態を中央で1つ持ち、通常モードは完全 no-op

`state.demo = { active, virtualDay }` を導入。`active` のときだけ上部バーにデモ帯＋日付コントロールを出し、目標/今日/振り返り画面のデータ取得先を `/api/demo/*`＋`virtualDay` に切り替え、作成・削除ボタンを無効化/非表示、今日タブはダミーパスワードを表示する。`active=false`（通常モード）では一切の分岐が既存経路のまま＝挙動不変。入り口は設定タブ内トグル（`デモを開始/終了`・`サンプルをリセット`）。

## Risks / Trade-offs

- **[本番 reveal がクライアント日付 `b.date` を受ける]** → 既に「その日が実記録で UNLOCKED のときだけ返す」ため未来日で達成は捏造不可。デモはこの経路を一切通さない（ダミー）。本変更で reveal のガードは触らない。念のため apply の受け入れ確認に「デモで未来へ飛ばしても本番 reveal 出力が不変」を含める。
- **[レポート描画コードの再利用でデモ専用の見た目差分が必要]** → デモ帯・ダミー表記・作成/削除の無効化のみを条件分岐で足し、レポート本体の4ブロック描画は共有コンポーネントを流用（別物 UI 事故を避けるため既存描画を分岐で包む）。
- **[seed とスキーマのドリフト]** → デモ DB は本番と同じマイグレーションを流してから seed するので、スキーマは常に本番と一致。焼き込み列（per_condition_results）の形が変わったら seed も追随が必要 → seed は1ファイルに集約し、変わりやすい形を1箇所に閉じる。
- **[インメモリ DB の常駐メモリ]** → サンプルは1目標30日分と小さく無視できる。サーバ再起動で消えて次回 `デモ開始`/`reset` で再構築されるだけ（永続不要）。

## Migration Plan

- 追加のみ（新ルータ・seed モジュール・フロントのデモ分岐）。本番 API・本番 DB スキーマ・reveal は変更しない。ロールバックはデモ関連の追加を外すだけで通常モードは完全に元通り。
- DB マイグレーション追加は不要（デモ DB は既存マイグレーションを流用）。
