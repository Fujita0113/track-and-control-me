## Context

振り返りタブ（`reflection.js`）は現在、左に Markdown エディタ＋気分ピル＋目標日記、右レールに対象日入力と過去一覧を持つ。データ可視化は無い。可視化資産は他所に散っている：

- `today.js` … 総作業 KPI・グループ別ドーナツ（分母＝総作業）・7日棒。Chart.js（`chart.umd.min.js`）を使用。
- `timeline.js` … 縦型・編集可能な詳細タイムライン（`GET /api/timeline/:date` → `{window,auto[],manual[],gaps[],splitOverrides[]}`）。

サーバ側の既存事実：
- `getTimeline(db, dayKey)` は `window.start/end` を **日境界**（`boundaryStartOfDay` / `day_boundary_minutes`）で返す。実際の起床時刻ではない。
- 作業 AUTO は `session.credited_ms`（同時オープン ÷N 済み）、自己申告 MANUAL は `creditedSeconds = (end-start)/n/1000`（`timeline-coactive-record` / issue #28）。持ち分は「各瞬間を co-active 数で等分」する分割なので、**覆われた全区間の持ち分合計 ＝ その区間の実時間**。
- `computeGaps` は `[winStart, capEnd]` で未カバー区間を返す（`capEnd = min(winEnd, max(now, winStart))`）。

issue #17 のユーザー意図：振り返り中に「一日の配分（円）」と「タイムライン（一望）」を見ながら、時間帯ごとの過ごし方を反省したい。ただし詳細タイムラインの常設は窮屈 → 「一望は軽い常設・詳細はオンデマンド」で解く（explore で合意した A＋B 合わせ技）。

## Goals / Non-Goals

**Goals:**
- 振り返りタブのエディタ上部に「一日の配分ドーナツ」を常設し、時系列は「右オーバーレイのテキストタイムライン（トグル）」で提供する。
- 円が端〜端でちょうど閉じる、持ち分ベースの正しい配分集計をサーバ側に一元化する（#28 の持ち分をそのまま流用）。
- 既存の振り返りエディタ・保存・過去一覧、タイムラインタブの記録挙動を一切変えない（`timeline.js` は無改修）。

**Non-Goals:**
- 覚醒時間の厳密化（起床時刻の手入力・睡眠計上）。MVP は「記録の端〜端」を覚醒時間の近似とする（Open Questions）。
- 振り返りタブでのグラフィカルな詳細タイムライン（縦型ビュー）の常設・引き出し。詳細な記録・編集はタイムラインタブに委ねる。
- デモモード（`demo-mode`）での配分表示。読み取り専用プレビューは後続。
- 週次・複数日比較・気分との相関などの発展的可視化。
- テキストタイムラインからの記録・編集（読み取り専用に限定）。

## Decisions

### D1: 分母は「記録の端〜端」（覚醒時間の近似）
アプリは起床/睡眠を知らないため、母数を **最初の記録の開始〜最後の記録の終了（当日は現在時刻を上限）** とする。日境界先頭・末尾の空白は母数から除外。理由：4:00 境界起点だと未記録が朝晩の巨大な空白に支配され、円が「空白だらけ」で反省に使えない。端〜端に絞れば「動いていた一日の中の配分」になり、内側の未記録＝離席・記録漏れとして意味を持つ。起床時刻手入力は将来拡張（D6/Open Questions）。

### D2: 配分は持ち分（credited）で集計 → 円がちょうど閉じる
各スライス秒 = Σ(AUTO `credited_ms`) ／ Σ(MANUAL `span/n`) をグループ／カテゴリ別に合算。未記録 = 端〜端の内側の gap 秒。÷N が各瞬間を過不足なく分割するので **Σスライス持ち分 ＋ 未記録 ＝ 端〜端総秒**。二重計上せず、`timeline-coactive-record` の評価規則と一致。並列作業がある日でも円が閉じる。

### D3: 集計はサーバ側の新サービス＋読み取り API
`server/src/services/` に `getDayAllocation(db, dayKey, nowMs?)` を新設し、`GET /api/timeline/:date/allocation`（`timeline.ts`）で公開。返却例：
```
{ dayKey, extentStart, extentEnd, totalSeconds,
  slices: [ { key, label, color, kind: 'WORK'|'MANUAL', seconds } … 降順 ],
  untrackedSeconds }
```
実装は `getTimeline` の credited／gap ロジックを再利用（session を credited_ms 集計、activity_log_entry を span/n 集計、端〜端 = min(first start)〜max(last end, 対象日=当日なら now)、その内側で `computeGaps`）。クライアント再計算より正確・一元化でき、splitOverrides や境界の扱いをサーバに閉じ込められる。api.js に `getAllocation(date)` を追加。

### D4: タイムラインは右オーバーレイのグラフィカル縦帯（読み取り専用・既存の短縮版）
**改定（ユーザーフィードバック）**：当初はテキスト行リストで代替する方針だったが、`開発 ＋振り返り` が細切れで延々と並ぶだけで「欲しい情報が無い」ため、**グラフィカルな縦帯（既存タイムラインの短縮版）** に変更する。元データは `GET /api/timeline/:date` の `auto/manual/gaps`（読み取り専用消費）。整形：
- 記録の端〜端に絞り（先頭・末尾の境界空白は出さない）、重なる記録をクラスタ（同時作業）へまとめる。
- 各ブロックを **左＝持続時間に比例した高さの色帯／右＝時刻レンジ＋ラベル** で描画。
- 同時記録（co-active / co_record）は **1本の帯を斜め縞（`repeating-linear-gradient`）** で多色表現（青×紫など）。`layout()` の列分割は使わない。ラベルは `＋` 併記。
- gap（`away_min_seconds` 以上）→ 中立色の帯＋`（未記録）`。
- **連続する同一構成の細切れブロックは1つに結合**（閾値未満の細切れギャップを橋渡し）してコンパクト化。

パネルは `rf-tlpanel`（`position: fixed`、右端オーバーレイ、`translateX` で出し入れ）。**既定は開**（issue #17: 初めは開いていてほしい／閉じられるようにはする）。トグル `rf-tl-toggle` は **目立つアクセント色＋アイコン＋シェブロン**（閉じている時ほど鮮やかで再発見しやすく、開いている時はグレーで「閉じる」を示す）。パネルは **前面に重なるオーバーレイ**（本文レイアウトは不変・第3カラム化しない）。開くと右レール（対象日・過去一覧）の上に浮いて重なる（レールを見たい時は閉じる。※当初は本文を左へ寄せてレールを露出させたが「3列に見える」ため撤回＝オーバーレイに戻した／issue #17）。読み取り専用（記録・編集動線なし）。詳細な記録・編集は従来どおりタイムラインタブへ誘導。

### D5: 縦帯は `reflection.js` の新規・独立実装（`timeline.js` は無改修）
グラフィカル縦帯は `timeline.js` の `layout()`／編集ビューを再利用せず、`reflection.js` 側に**新規の軽量ロジック**（`buildRibbon`：クラスタ化・端〜端・同一構成結合・縞生成）として実装する。`timeline-run-view` は不変・タイムラインタブ専用のまま（`timeline.js` への改修ゼロ＝退行リスクなし）。編集動線を持たない点は当初方針から不変。

### D6: 配分はドーナツではなく横棒リスト（純 DOM/CSS・Chart.js 不使用）
**改定（ユーザーフィードバック）**：未記録が最大スライスになると円が「未記録だらけ」で気が滅入り、情報として薄い。ドーナツ（Chart.js）をやめ、**横棒リスト**にする：
- 各行＝ラベル＋持ち分秒に比例した横棒（トラック内 fill）＋時間表記。作業＝グループ色／自己申告＝色／未記録＝中立色。
- **未記録以外は時間降順に上から、未記録は常に最下部固定**（未記録が最大でも先頭に来ない）。
- Chart.js に依存せず純 DOM/CSS で描画（`window.Chart` 不使用・軽量・CSP 安全）。

レイアウトと DOM 配置：
- 配分バー `rf-alloc`：**本文エディタカード（`rf-card`）の直上**（気分ピルの下・エディタの上）にコンパクト常設。
- 縦帯タイムライン `rf-tlpanel`：`position: fixed` 右オーバーレイ。トグルボタン `rf-tl-toggle` をタイトル行右に置く。
- 縦帯の色帯はインライン style（`element.style.background`／CSSOM）で単色・縞を適用。その他スタイルは `rf-*` クラス＋CSSOM（インライン style 属性なし＝CSP 準拠）。
- 破棄：タブ離脱時にパネルを破棄（`hide()`）。対象日切替時はバー・（開いていれば）縦帯を対象日で再構築。

### D7: 対象日追従
`loadEditorForDate(date)` の末尾で配分・帯を再描画（`renderDayOverview(date)`）。日付入力変更・過去選択の既存動線（いずれも `loadEditorForDate` を通る）にそのまま乗る。開いている引き出しがあれば対象日で再マウント。エディタ本文のロードとは独立に失敗しても本文編集を妨げない（`catch` で握り、空/エラー状態表示）。

## Risks / Trade-offs

- **端〜端の分母は「起きてすぐ／終えてすぐ」記録がある前提に弱い**。朝の空白は母数外なので、実際は起きていたのに記録が無い時間は見えない。MVP の割り切り。将来の起床時刻手入力（D1）で厳密化可能。
- **円の豊かさは記録の埋まり具合に依存**。休憩を自己申告していない日は未記録スライスが大きくなる。これは欠点であると同時に「タイムラインを埋めよう」を促すフィードバックにもなる（explore で確認済みの意図）。空状態・大きな未記録は正直に見せる。
- **テキスト行での並行作業の簡約**：同時記録が多い区間は `＋◯◯＋△△` と併記が長くなる可能性。振り返りの概観としては許容し、細かな並列の可視化はタイムラインタブに委ねる。
- **`timeline.js` は無改修**：詳細ビュー再利用をやめたため、タイムラインタブへの退行リスクは実質なし。テキスト整形は `reflection.js` 側の新規・独立ロジック。
- **フェッチ数**：ドーナツは allocation API（常設）、テキストタイムラインは開いた時のみ timeline API を対象日で取得（トグルなので閉じている間は不要）。いずれも軽量で許容。

## Open Questions

- **起床時刻の手入力**（覚醒時間の厳密な分母）は別 issue とするか。#17 の「起きてから」を厳密に満たすには必要だが、MVP は端〜端近似で進める。
- **デモモードの配分**：`demo-mode` に配分ドーナツ／帯を出すか。読み取り専用プレビューとして後続で追加候補。
- **配分の粒度**：作業をグループ別スライスにするか「作業」1 スライスに畳むか。初期はグループ別（既存色資産を活かす）。視認性次第で「作業まとめ＋休憩内訳」への切替を検討。
