## Context

タイムライン画面は ui-revamp で Google カレンダー風（`timeline.js`、単一縦カラム・時刻ガター・列分割）に刷新済みだが、実運用で可読性の問題が露呈した。

- サーバーは同一グループの近接セッションを `session_coalesce_seconds`（既定 120s）で coalesce するため、2分を超える離席のたびに AUTO ブロックが分断され、`blockEl()` が断片ごとにタイトルを描く（「面接」が上下に連呼）。
- `boundaryMinutes()` が全クラスタ境界の時刻ラベルをガターへ絶対配置するため、断片が多い日はラベルが衝突・重なりする。
- サーバーの `getTimeline()` は未カバー区間 `gaps`（現状 ≥300s 固定）を計算済みだが、フロントは一切描画していない。離席の記録手段は空き領域のドラッグのみ。
- ユーザーは作業中ダッシュボードを開かない運用。離席復帰の瞬間を知るのは拡張の SW（`chrome.idle.onStateChanged` 購読済み、`Snapshot.lastActiveTs` 保持済み）だけ。
- ユーザーの意味論確認済み: 同時に開いているグループは「お互いの待ち時間に並行で進めている」ものでありメイン/サブは存在しない。並列カラム表示と divide-by-N クレジットは正しく、維持する。

関連する既存部品: `POST /api/timeline/:date/gap-to-away`、カテゴリチップ式の記録ポップオーバー（`openDraft`）、WS ハンドシェイクの `welcome` メッセージ、拡張の `WsConfig.wsPort`。

## Goals / Non-Goals

**Goals:**
- 「何時から何時に何をしていたか」が一目で読めるタイムライン（ラン描画・ガター簡素化・ラベル1回表示）
- 未記録の離席がゼロクリックで見え、ワンクリックで記録できる（ゴーストスロット）
- 離席復帰のたびに記録を促す OS 通知（ダッシュボード非表示前提）
- 「吸収 / 記録すべき離席」の境界を単一閾値に一元化し、取りこぼし帯を構造的に排除

**Non-Goals:**
- 集計・クレジット計算（divide-by-N、gapCap、day boundary）の変更
- session / activity_log_entry スキーマの変更、`session_coalesce_seconds` の値変更
- アクティブグループの主役化・並列カラムの廃止（意味論確認により不採用）
- アジェンダ（リスト）ビューの追加、`ungrouped` ブロックの表示方針変更、モバイル対応

## Decisions

### D1. ラン結合は表示レイヤー（クライアント）のみで行う
同一 `stableGroupId` の隣接 AUTO ブロック a, b を、次の**両方**を満たす場合に限り1つの「ラン」として描画する:
1. `b.startAt - a.endAt < away_min_seconds * 1000`
2. 区間 `(a.endAt, b.startAt)` に他の描画ブロック（他グループの AUTO / MANUAL）が一切重ならない

条件2がないと「面接→ブログ5分→面接」の切り替えまで結合され、ブログ作業時間が面接ランの離席ハッチとして描かれて嘘になる。MANUAL（記録済み離席）が挟まる場合も同様に結合しない（ブロックの重なり描画を防ぐ）。

ランは `{ title, color, startAt, endAt, segments[], innerGaps[], creditedMs(合計), coactiveKeys(和集合) }` を持つ描画専用の構造体で、データ層には存在しない。

**代替案と棄却理由**: (a) `session_coalesce_seconds` を10分へ引き上げ → `computeGaps` が coalesce 後ブロックを参照するため、ゴースト/通知に使うギャップデータ自体が消える。データの正直さも失う。 (b) サーバー側に run API を追加 → 表示専用の概念をサーバーへ持ち込む必要がなく、閾値変更時の再取得も不要なため過剰。

### D2. 閾値 `away_min_seconds` の一元化（既定 600s）
新しい app_config カラム `away_min_seconds INTEGER NOT NULL DEFAULT 600` を追加し、次の3箇所すべてが参照する:
- サーバー: `computeGaps()` のフィルタ（現状 300s ハードコードを置換）
- クライアント: D1 のラン結合条件（`state.config` 経由で取得）
- 拡張: 復帰通知の最小離席時間（D7 の welcome 配布で取得）

「閾値未満＝作業の呼吸として吸収、以上＝記録すべき離席」という単一ルールになり、どこにも現れない帯（例: ラン結合されず、ゴーストにもならない7分）が構造的に消える。設定画面には分単位で表示・編集（保存は秒）。契約 `DEFAULTS` に `AWAY_MIN_SECONDS: 600` を追加し拡張のフォールバックとする。

### D3. ハッチスライス表現と最低視認性
ラン内部の間隙（`innerGaps`）はブロック内のオーバーレイ（`repeating-linear-gradient` の斜線、半透明白）で描く。PXM=1.2px/分では8分の離席が約10pxにしかならないため:
- スライスの最低描画高さを 4px 保証（実時間比例が原則、下限のみ補正）
- スライス高さ ≥ 16px の場合のみ「離席 n分」ラベルをスライス内に表示、未満は `title` 属性のツールチップ
- 正確な内訳（離席 n回・合計 m分・各区間）は詳細ポップオーバーに一覧表示

### D4. 「・同時n」サフィックス廃止とクレジット内訳のポップオーバー移動
ブロック名は `title` のみ（横並びカラム自体が同時性を伝える）。AUTO ラン/ブロックの詳細ポップオーバーに以下を追加:
- 時間帯（ランのスパン）と実働（クレジット）: 断片 `creditedMs` の合計を `fmtDur` で表示
- 同時に開いていたグループ: `coactiveGroupKeys` の和集合をグループ名で列挙（「この時間帯は同時オープンのため均等割」の注記）
- 離席内訳（D3）

n は区間ごとに変動しうるため「同時n」という単一数値では正確に表せない — サフィックス廃止はノイズ削減と正確性の両方に資する。

### D5. ガターは正時ラベルのみ、境界目盛り・境界破線は廃止
`boundaryMinutes()` とその出力（`tlc-tick-lbl`、`tlc-boundary-line`）を削除する。正確な開始/終了はラン自身の時刻ラベル（`HH:MM – HH:MM`）と詳細ポップオーバーが担う。ガターの情報密度は Google カレンダー本家と同じ「正時のみ」になる。

### D6. カラム割当の安定化（前回カラム優先の first-fit）
`layout()` のクラスタ分割・first-fit は維持しつつ、レンダー内で `stableGroupId → 直前クラスタで使ったカラム index` のマップを持ち、空いていればそのカラムへ再割当、衝突時のみ first-fit へフォールバックする決定的アルゴリズムにする。ラン化でクラスタ境界自体が激減するため、この軽量な優先規則で「同一グループが左右へフリップする」現象は実用上解消される見込み。

### D7. 復帰検知は拡張 SW、閾値は welcome で配布
- **復帰検知**: `emitSample()` 内で `prev.idleState !== 'active' && gathered.idleState === 'active'` の遷移を検出し、`now - prev.lastActiveTs >= 閾値` なら通知。スリープ/ブラウザ再起動は `bootstrap()` / `onStartup` 時に同じ判定を行い補足する。`Snapshot` に `lastAwayNotifiedTs` を追加し、同一離席区間への重複通知を抑止する。
- **通知**: `chrome.notifications.create`（basic）。タイトル「離席を記録しませんか？」、本文「HH:MM – HH:MM（n分）離席していました」。本体クリックで `http://127.0.0.1:${wsPort}/#timeline?from=<ms>&to=<ms>` を `chrome.tabs.create` で開く。`manifest.json` の `permissions` に `"notifications"` を追加。
- **閾値配布**: WS `welcome` メッセージに `awayMinSeconds`（optional）を追加し、拡張は受領時に `chrome.storage.local` へ保存。未受領（未接続・旧サーバー）時は `DEFAULTS.AWAY_MIN_SECONDS`。optional フィールド追加のため旧クライアントとも後方互換。
- **離席開始時刻の定義**: 通知の区間は `lastActiveTs → 復帰時刻`。正確な区間の権威はあくまでサーバーのギャップ計算であり、通知の時刻は案内用（プリフィル値）とする。

**代替案と棄却理由**: (a) ダッシュボードのバナー/トースト → 作業中ページを開いていない運用のため届かない。 (b) 拡張が REST API をポーリング → SW のライフサイクルと相性が悪く、WS 経由の配布で十分。

### D8. ディープリンクは hash パラメータ、フロントで消費
現状ルーティングは JS タブ切替のみで URL を使っていない。`main.js` の `boot()` で `location.hash` が `#timeline` で始まる場合に timeline タブを activate する最小限のハッシュ対応を入れる。`timeline.js` の `show()` は `from`/`to` パラメータを読み取り、該当日（`from` から日付を導出。day boundary 04:00 を跨ぐ精密な dayKey 導出はフロントの既存 `state` 情報で行い、導出不能時は `state.today`）を表示したうえで記録ポップオーバーを区間プリフィルで自動オープンし、`history.replaceState` でパラメータを消費（リロードで再発火させない）。拡張は `from`/`to` のみ渡し、dayKey 計算をフロントに寄せる。

### D9. ゴーストスロットとドラッグの共存
`tl.gaps`（D2 により ≥ 閾値のみ）をレーンに破線アウトラインの「＋ 未記録 HH:MM–HH:MM（n分）」として描画。クリックで既存 `openDraft(start, end)` を区間プリフィルで開く（確定 API は既存 `addManual` のまま）。ゴースト要素は `mousedown` を `stopPropagation` してドラッグ記録と干渉しない。空き領域ドラッグは存置し、`gapContaining()` はラン全スパン（ハッチ含む）を占有として扱う。凡例・ヒント文言をゴースト前提に更新する。

## Risks / Trade-offs

- [ランのスパン表示が「連続稼働」に見える] → ハッチスライス＋ポップオーバーの実働（クレジット）/離席内訳で緩和。ラベルはスパンを表示し、実働は必ずポップオーバーで確認できる。
- [通知の過剰発火（頻繁な短時間離席）] → 閾値未満は通知自体を出さない。`lastAwayNotifiedTs` で重複抑止。それでも煩ければ設定で閾値を上げる運用（閾値は D2 の単一値なので表示側も一貫して変わる）。
- [ギャップ閾値 300s→600s の挙動変更で、既存の5〜10分ギャップが API から消える] → 意図した仕様変更（同一グループならランのハッチとして可視、異なるグループ間なら小さな空白として残る）。spec に明記。
- [SW が停止中で idle 遷移イベントの発火が遅れる/落ちる] → `chrome.idle.onStateChanged` は SW を起こすため概ね検知可能。取りこぼしても heartbeat 再開時・bootstrap 時の判定で補足。通知はあくまで補助経路で、ゴーストスロットが確実な回収経路（二段構え）。
- [サーバー停止中に通知をクリックするとページが開けない] → 通知は即時記録のショートカットと割り切る。サンプルは拡張側にバッファされるため、サーバー再開後にギャップはゴーストとして回収可能。
- [閾値変更が拡張へ反映されるのは WS 再接続時] → 反映遅延は許容（welcome 受領で更新）。即時性が必要なら popup の applyConfig 経由で再接続をトリガーできる既存導線がある。
- [カラム安定化はヒューリスティック] → 完全な安定性（日単位の固定カラム）は幅効率とトレードオフのため見送り。ラン化との相乗で実用上十分と判断。効果不足なら追補 change で対応。

## Migration Plan

1. contract に `AWAY_MIN_SECONDS` と welcome の `awayMinSeconds`（optional）を追加 → ビルド
2. サーバー: マイグレーションで `away_min_seconds` カラム追加（既存 DB は ALTER TABLE、既定 600）→ `computeGaps` / 設定 API / welcome 送信を対応
3. フロント: `timeline.js` 描画部の書き換え（ラン・ゴースト・ガター・ポップオーバー・ディープリンク）、`settings.js` に閾値項目、CSS 追加
4. 拡張: 通知ロジック＋ `notifications` 権限追加 → 拡張の再読み込み（バージョン更新）が必要
5. ロールバック: フロント/サーバーは前コミットへ戻すのみ。追加カラムは残置で無害（旧コードは参照しない）

## Open Questions

- 通知の頻度制御（毎回 vs 1日の上限）: まず「閾値以上は毎回」で開始し、運用で煩ければ追補で対応
- ハッチスライスの視覚密度（色・角度・透明度）: 実装時に実データでチューニング
