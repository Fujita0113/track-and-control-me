# Edge タブグループの「同一性」が壊れていた話 — stableGroupId バグと identity レジストリでの解決

- 対象 issue: #59
- OpenSpec change: `group-rule-snapshot-identity`
- 記録日: 2026-07-24

Zenn記事化のための下書きメモ。時系列・原因・設計判断・実装・運用上のハマりどころをまとめる。

## TL;DR

- 自作の「タブグループ単位で作業時間を計測し、解錠ルールに使う」Chrome/Edge拡張で、**グループの同一性（stableGroupId）の採番が壊れていた**。
- 原因は拡張側のフォールバックロジック `title + ' ' + color` が、**新規グループ作成直後の空タイトル状態にも適用**されてしまい、色が同じというだけで無関係な既存グループのUUIDを継承していたこと。
- 結果、実データで **116個のタブグループに対してUUIDが12個しか発行されておらず**、同時に開いている別グループ同士が同じIDを共有するサンプルが **2062件** あった。
- 表示系（グループ別内訳・タイムライン）は以前の改修で「記録時点の名前＋色」ベースの分類にすでに移行していたため実害が少なかったが、**解錠ルールの評価だけが壊れた `stable_group_id` に依存**していたため、「競技プログラミングのグループを指定したはずのルールが、面接に費やした時間で解錠される」という実害が出ていた。
- 解決策は、拡張機能側のUUID採番に依存するのをやめ、**サーバー側に「記録時点の名前＋色」から解決する identity レジストリ**を新設し、表示・ルール判定・選択肢のすべてをそこに統一すること。あわせてタブグループの改名を検出してサーバーへ伝え、識別が改名をまたいでも壊れないようにした。
- 移行後も、**凍結ポリシー（当日ルールは編集不可）と「要再設定」表示の間で、当日中はどうにもならないケースが発生する**という運用上の落とし穴が見つかった（後述）。

## 症状（issue #59）

ゲート画面（今日タブ）の解錠条件が、`グループ: 70d5118e-e7c2-467d-8097-73a500a5e9bf` のような**生のUUIDで表示**されており、ユーザーからは何の条件か分からなかった。

調査を進めると、UUID自体の中身が壊れていることが判明した。実DBを見ると、`70d5118e…` という1本のUUIDが `英語` / `Python` / `webエンジニアリング` / `面接` / `振り返り` / `アルゴリズム` / `競技プログラミング` / `開発` という**まったく無関係な8つのグループを飲み込んで**いた。

これにより実害として、「競技プログラミングのグループを指定したはずのルールが、面接に費やした時間で達成扱いになる」という、ルールの意味そのものがすり替わる不具合が起きていた。

## 原因

### 拡張機能の `stableGroupId` 採番

Chrome/Edge拡張の `chrome.tabGroups` API が返す `groupId` は、ブラウザセッション内でのみ有効な揮発値で、再起動すると振り直される。これを安定させるため、拡張側で `chrome.storage` に UUID を永続化する `stableGroupId` マネージャーを実装していた。

```ts
// 揮発的な groupId(string) -> stableGroupId(UUID)
const GROUP_BY_ID_KEY = 'groupStableMap';
// "title color" -> stableGroupId(UUID)（再起動で groupId が変わっても引き継ぐ）
const GROUP_BY_IDENTITY_KEY = 'groupIdentityMap';

function identityKey(title, color) {
  return `${title} ${color}`;
}
```

`groupId` で引けなければ `(title, color)` の組み合わせ（フォールバック identity）で既存のUUIDを探し、それも無ければ新規採番する、という設計。

### バグの本体

**新規タブグループを作成した直後は一瞬 `title === ''`（空文字）になる。** このときフォールバック identity キーが `' pink'` や `' blue'` のような**色だけの文字列**になってしまい、たまたま同じ色で先に登録されていた**無関係な既存グループのUUIDを継承してしまう**。

さらに、名前入力ダイアログでその後に色を選び直すと、色をまたいで同じ現象が連鎖する。結果として、時間が経つにつれて多くのグループが少数のUUIDへどんどん収束していく。

これが「116グループ → 12 UUID」「同一瞬間に開いている別グループが同じIDを共有するサンプルが2062件」という実データの惨状につながった。

### 「直したはずなのに直っていなかった」問題

空タイトルガード自体は過去のコミット（`7b0f943`）で一度修正されていた。しかし、**7/22 に作られたグループが空タイトル時点で既存UUIDを継承していた**ことから、実際にブラウザで動いていたのは修正前のビルドだったと推定された。つまり「コードは直っていたが、拡張機能の再ビルド・再読み込みが漏れていた」。

これが後述する「拡張バージョンの警告バナー」を作るきっかけになった。

## なぜルール判定だけが実害を受けたのか

同じアプリの中で、実は3層のグループ同一性が併存していた。

1. **拡張機能の `stableGroupId`**（上記の壊れた層）
2. **表示系のスナップショット identity**（`tab_group_name_snapshot` + `group_color_snapshot`）— グループ別内訳・タイムライン・振り返りリボンは、以前の改修でこちらへ移行済みだったため、壊れた `stable_group_id` を回避できていた
3. **ルール評価**（`rule_condition.stable_group_id` と `daily_totals_snapshot` の `stable_group_id` 単位）— **ここだけが1のバグをそのまま被っていた**

つまり「表示は直っているのにルール判定だけおかしい」という食い違いが、そもそも issue の見え方をわかりにくくしていた。

## 設計判断

### D1. サーバー側に identity レジストリを置く（拡張のUUIDには依存しない）

新テーブル `group_identity`（現在名・現在色・作成/最終観測時刻）と `group_identity_alias`（`(name, color)` → identity）を新設。記録時点スナップショット `(name, color)` を**必ず**別名表経由で identity へ解決する。未知の組は identity を新規作成し、その組を別名として登録する。

なぜ拡張の `stableGroupId` を主キーにしないか: 揮発的な chrome `groupId` を跨いだ同一性の維持は本質的に推測であり、実際に壊れた。`(name, color)` は利用者が意図的に付けた識別子であり、`session` にすでに不変のスナップショットとして残っている。サーバー側で解決すれば、拡張のバグ・再インストール・別プロファイルにも耐えられる。

### D2. ルール条件は identity の内部ID（INTEGER）を参照する

`condition_key = 'group:<identityId>'`。判定は「その identity の別名すべてに一致する `session.credited_ms` の合算」で行う。

```sql
SELECT COALESCE(SUM(credited_ms), 0) FROM session
 WHERE day_key = ?
   AND (tab_group_name_snapshot, COALESCE(group_color_snapshot,'')) IN (<identity の別名集合>)
```

`session.credited_ms` は「グループ別内訳」が使っているのと同じ源泉なので、**内訳の数字とゲートの進捗が定義上一致する**ようになった。

なぜ `condition_key` に名前を埋め込まない（`group:<color>:<name>` にしない）か: 改名のたびに `rule_condition` / `goal_practice` / `daily_check` / `unlock_evaluation.per_condition_results` に散らばったキー文字列を書き換える必要が生じ、破綻しやすい。内部IDなら改名は `group_identity.name` の1行更新で済む。

### D3. タブグループの改名を検出してサーバーへ伝える

拡張が `tabGroups.onUpdated` の title/色変更を検出し、静止5秒のデバウンス後に `(旧名,旧色)→(新名,新色)` を1件の `groupRename` メッセージとして送る（`ActivitySample` とは別メッセージ）。名前入力は1文字ごとに更新イベントが発火するため、デバウンスで確定後の変化だけを送る。

デバウンス中にService Workerが停止する（MV3のSWは約30秒でアイドル停止する）ケースに備え、保留中の改名は `chrome.storage` に永続し、次回のSWウェイク時に必ずフラッシュする。

サーバーは改名を受けたら1トランザクションで: identityの現在名/現在色を更新 → 旧名を別名として保持 → 新名を別名登録 → 新名が既に別identityに属していれば統合（作成が古い方を残す）→ 採用中の目標実践のラベルスナップショットを更新。

改名前の区間も同じidentityに解決され続けるため、**改名当日の進捗が0に巻き戻らない**。

### D4. 拡張機能の採番を不変条件で固める

1. 空タイトルのとき identity フォールバックを一切引かない・書かない
2. 同一解決パス内で2グループが同じ `stableGroupId` を持ってはならない（重複を検出したら、先に観測された方=`groupId`が小さい方が既存IDを保持し、他方を再採番）
3. 写像のスキーマバージョンを上げて（2→3）、汚染済みマップを強制クリア
4. 拡張バージョンをサンプルへ載せ、サーバーが既知の最小要求版未満のビルドを検出したらダッシュボードに警告バナーを出す（「修正したのに反映されていない」を二度と無警告で見逃さないため）

### D8. 後方互換（旧 `group:<uuid>` 条件）

既存データは書き換えない。過去のセッションは記録時点の `(名前, 色)` から identity へ解決する。identity参照を持たない旧条件は**従来どおり `daily_totals_snapshot` の `stable_group_id` 単位で評価**し、過去の判定結果を変えない。表示だけ `tab_group` の名前で解決し「（要再設定）」ヒントを添える（UUIDは出さない）。

## 実装の要点（ダイジェスト）

- **DBマイグレーション**: `group_identity` / `group_identity_alias` を新設。既存 `session` の distinct `(名前, 色)`（空名・未グループを除く）から identity を初期構築（既存行は書き換えない）。`rule_condition` に `group_identity_id` を追加（既存の `stable_group_id` 列は残す）。
- **`services/group-identity.ts`**: `resolveIdentity` / `renameIdentity`（統合込み）/ `listRecentGroupIdentities` / `resolveGroupDisplay` を実装。表示系（`summary.ts` / `timeline.ts`）は共通の `loadIdentityResolver` を介して同一ロジックを共有する。
- **`rules/evaluate.ts`**: `GROUP` 条件の判定を identity 参照ベースに変更しつつ、`group_identity_id` を持たない旧条件は従来経路のまま評価する後方互換分岐を残した。
- **UI**: 今日タブ・ルール一覧・目標画面のGROUP条件表示をUUIDから「現在名＋色チップ」に変更。グループ選択肢は壊れた `tab_group` テーブルではなく `GET /api/groups/recent`（直近30日に実測された identity・合計時間降順）から出す。
- **拡張機能**: `GROUP_MAP_SCHEMA_VERSION` を3へ引き上げて汚染マップを強制クリア。改名検出＋デバウンス送出。`manifest.json` のバージョンを上げ、サーバー側に最小要求版チェック＋警告バナーを追加。
- **デモモード**: 既存サンプルに「登録済みの改名」（例: グループ名A→Bへ改名しても進捗が続く）シナリオを追加し、実サーバー経由で確認。

## 移行後に見つかった運用上のハマりどころ

修正を入れて実際にダッシュボードを見ると、当日の解錠条件に「グループ: 面接（要再設定）」という表示が出た。ユーザーの認識では「もともと競技プログラミングのグループで登録していた」条件。

**原因**: これは移行前に作られた旧形式の条件（`group:<uuid>`、`group_identity_id` が NULL）。当時「競技プログラミング」のつもりで選んだグループが、内部的には壊れたUUIDに紐づいており、そのUUIDが**現在たまたま「面接」という名前の `tab_group` 行を指している**ため、この表示になっていた。

**問題**: 「（要再設定）」と表示されるのに、**当日のルールは凍結ポリシーにより既存条件を編集・削除できない**（当日追加できるのは新しい条件のみ）。つまり当日中はこの表示を見ても何もできず、ユーザーを困惑させる詰み状態になっていた。

これは今回の実装のバグというより、**「凍結ポリシー」と「要再設定という気づきのための表示」が組み合わさったときに生まれるUXの落とし穴**だった。「気づけるようにする」ことと「その場で直せるようにする」ことは別問題であり、後者は当日は原理的に無理（凍結は「当日中に既存の解錠条件を骨抜きにするゲーミングを防ぐ」ためのポリシーで、これを緩めると凍結の意味自体が壊れる）。

**対処**: 翌日以降のルール編集で、グループ選択を正しく選び直せば、新しい identity 参照を持つ条件として保存され、以後は正しく動く。ただし、その条件が30日チャレンジの目標（ジャンル固定）に採用されている場合、`condition_key` が変わる（`group:<uuid>` → `group:<identityId>`）ため、単純な差し替えだと目標側のロックに引っかかる可能性があり、目標側の対応もあわせて必要になる。

## 教訓（Zenn記事の切り口候補）

- **フォールバック同一性ロジックは「初期状態（空・未確定）」の扱いを最初から設計しないと、時間とともに収束・崩壊する**。今回は「新規作成直後の空タイトル」という一瞬の状態が、識別子の生成ロジックに侵入してしまった。
- **「表示は直っているのにロジックだけ古い参照を使っている」状態は気づきにくい**。同じ「グループ」という概念に対して、モジュールごとに異なる同一性の定義を持ってしまうと、部分的な移行が新たなバグの温床になる。
- **「ビルドし直したのに直っていない」は、コード修正そのものより発見が難しい**。バージョンを可視化し、古いビルドを警告する仕組みを持つべき。
- **ユーザー向けの「要再設定」的なヒントは、実際にその場で再設定できるかどうかを必ず検証する**。凍結・ロックなどの他のポリシーと衝突すると、「気づけるが直せない」という中途半端なUXになる。
- **拡張機能側の揮発的なIDに頼らず、サーバー側で意味のある単位（今回は「記録時点の名前＋色」）から同一性を再構築する**というアプローチは、クライアントのバグに強い設計になる。
